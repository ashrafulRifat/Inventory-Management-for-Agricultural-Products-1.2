This version is prepared for viva presentation. It uses clearer narrative flow, stronger role-based explanation, and explicit mapping between pages, database entities, and CRUD behavior.

## 1. Viva Quick Start

### 1.1 One-line project summary
This system digitizes agricultural operations from field activity to procurement, storage, IoT monitoring, and quality assurance using a relational MySQL database and role-based PHP web modules.

### 1.2 Core tech stack
- XAMPP (Apache + MySQL/MariaDB)
- PHP 8 with PDO prepared statements
- HTML/CSS/JavaScript
- Session-based authentication and role authorization

### 1.3 Main runtime pages
- Dashboard Home
- Field Operations
- Product Catalog
- Supplier Management
- Inventory and Storage
- IoT Sensor Network
- Quality Control

### 1.4 Role matrix (who uses what)

| Role | Main Responsibility | Primary Pages |
|---|---|---|
| Admin | Full cross-module oversight | All modules |
| Field Officer | Field data capture and request generation | Field Operations, Sensors (field scope), Dashboard |
| Inventory Manager | Product, supplier, storage, stock operations | Products, Supplier Management, Inventory, Sensors, Dashboard |
| Supplier | Order fulfillment visibility and status updates | Supplier Management (scoped), Dashboard |
| QC Officer | Quality inspection and spoilage tracking | Quality Control, Dashboard |
| IoT | Sensor and telemetry administration | Sensors, Dashboard |

## 2. Shared System Architecture (Presentation View)

### 2.1 Common page shell
Every module page follows the same structure:
- Left sidebar (role-filtered navigation)
- Top header (user identity and role)
- Main content area (cards, forms, tables)

Implemented through shared includes:
- includes/header.php
- includes/sidebar.php
- includes/footer.php

### 2.2 Authentication and authorization
- Session starts in includes/auth.php.
- Login checks users table and password hashes in auth/login.php.
- Route protection:
  - require_login() blocks anonymous access.
  - require_roles() enforces role-level module access.
- Admin access is globally permitted by has_role() logic.

### 2.3 Database connectivity
- config/database.php defines get_pdo() singleton.
- PDO settings:
  - exceptions enabled
  - associative fetch mode
  - emulated prepares disabled
- Security model:
  - Prepared statements in write operations
  - CSRF tokens for form submission
  - Escaped output using helper h()

### 2.4 UI interaction helpers
- Flash success/error messaging from session
- Delete confirmation dialogs via assets/js/app.js
- Dynamic line-item row add/remove in Supplier module
- Status-to-color badge mapping through helper badge_class()

## 3. Database-Centric CRUD Pattern (Used Across Modules)

### 3.1 Standard operation flow
1. User submits a form with action value.
2. Server validates CSRF token.
3. Server validates required and numeric fields.
4. Server checks role authorization.
5. Server runs prepared INSERT/UPDATE/DELETE query.
6. Server stores flash message.
7. Server redirects back to module route.
8. Module reloads with fresh SELECT results.

### 3.2 Foreign key handling pattern
- Forms use dropdowns for FK references.
- Tables show joined business labels (names, product + storage) instead of raw IDs.
- Deletes are naturally constrained by FK rules. Dependent rows block unsafe deletion.

### 3.3 Display IDs vs real IDs
- Database primary keys remain integer/bigint as designed in ER schema.
- UI display codes (FO-xx, PRD-xxxx, STK-xxxx, etc.) are generated by helper display_code().

## 4.1 Dashboard Home

### A. Basic overview
- Module file: modules/dashboard.php
- Route: /agri_inventory_management/modules/dashboard.php
- Main purpose: Unified operational monitoring page across all business domains.
- Main users: All authenticated roles.

### B. UI structure
- Metric cards:
  - Pending Input Requests
  - Low Stock Alerts
  - Active Sensor Deployments
- Monitoring tables:
  - Recent Purchase Orders
  - Recent Quality Inspections
  - Recent Telemetry Logs
  - Environment Condition Alerts

### C. Functional behavior on load
- Reads live metrics from:
  - input_requests
  - inventory_stock
  - sensors_registry
  - location_environment_status
- Fetches recent transactional/monitoring records with joins:
  - purchase_orders + suppliers
  - quality_inspections + qc_officers + inventory_stock + products + storage_facilities
  - sensor_telemetry_logs + sensors_registry + fields/storage_facilities
  - location_environment_status + fields/storage_facilities
- Uses ORDER BY descending and LIMIT for concise dashboard windows.

### D. CRUD behavior
- Read-only module (no forms for create/update/delete).

### E. Backend logic
- require_login() only
- try/catch around dashboard query block
- Uses shared helpers:
  - display_code() for visual IDs
  - badge_class() for status badge colors

### F. Business logic emphasis
- Low-stock rule is explicitly computed from stock thresholds.
- Environment alerts focus on monitor/critical rows for quick intervention.

### G. ER relationship emphasis
- This page is the best ER-integration showcase because it visualizes many relationships simultaneously:
  - purchase_orders -> suppliers
  - quality_inspections -> inventory_stock -> products + storage_facilities
  - telemetry_logs -> sensors_registry -> fields/storage_facilities
  - location_environment_status -> fields/storage_facilities

### H. Viva user journey example
1. Login and open Dashboard.
2. Explain live KPI cards as query outputs, not static values.
3. Drill into recent orders and inspection context.
4. Show telemetry and alert rows as operational risk signals.

### I. Navigation role
- Monitoring hub that links to all operational modules via sidebar.

### F. Business logic emphasis
- Captures the origin of supply-demand data:
  - harvest progress
  - input demand from field activity

### G. ER relationship emphasis
- Major ER links represented directly:
  - farmers -> fields
  - field_officers -> fields
  - fields -> harvest_logs

### H. Viva user journey example

- Core relationships used:

## 4.6 IoT Sensor Network

### A. Basic overview
- Module file: modules/sensors.php
- Route: /agri_inventory_management/modules/sensors.php
- Main purpose: Sensor administration, telemetry ingestion, and environment condition maintenance.
- Main users:
  - IoT: full control
  - Field Officer: field-scoped visibility and telemetry write
  - Inventory Manager: telemetry write and monitoring
  - Admin: full access

### B. UI structure
- Environment Status section
- Sensor Registry section
- Recent Telemetry section
- Forms shown conditionally based on role and action permissions

### C. Functional behavior on load
- Reads fields and storage lists for location selection
- Reads sensors with joins to location names
- Reads telemetry logs joined with sensor and location context
- Reads environment status rows with location context
- Field officer scoping:
  - sensor/environment data limited to fields overseen by that officer

### D. CRUD behavior
- IoT full CRUD:
  - sensors_registry
  - sensor_telemetry_logs
  - location_environment_status
- Non-IoT telemetry write path:
  - create telemetry entries (role-permitted)
- Validation:
  - sensor category whitelist
  - overall condition whitelist
  - at least one location (storage or field)
  - numeric telemetry value
  - datetime-local to SQL DATETIME normalization

### E. Backend logic
- Key local functions in module:
  - evaluate_temperature_status()
  - evaluate_humidity_status()
  - recompute_environment_status()
- Recompute algorithm:
  1. Identify sensor location
  2. Read latest temperature and humidity values at that location
  3. Compute temperature/humidity status
  4. Derive overall condition (secure/monitor/critical)
  5. Update existing environment row or insert new one

### F. Business logic emphasis
- Telemetry is not passive storage; it triggers condition intelligence.
- Critical/monitor status supports proactive intervention for crop/storage safety.

### G. ER relationship emphasis
- Main graph:
  - sensors_registry -> sensor_telemetry_logs
  - sensors_registry -> storage_facilities or fields
  - location_environment_status -> storage_facilities or fields
- Both location-capable entities enforce at least one location selected.

### H. Viva user journey example
1. Register humidity sensor for a storage facility.
2. Add telemetry reading.
3. Show automatic environment-status recomputation.
4. Demonstrate role-scoped visibility for field officer.

### I. Navigation role
- Hybrid module: monitoring plus real-time operational data ingestion.

## 6. Database Integration Summary

### 6.1 How frontend connects to database
- Each module uses get_pdo() from config/database.php
- SQL is executed directly in module files using prepared statements

### 6.2 How IDs and joins are used
- Primary keys are numeric (ER-compliant)
- Foreign keys are selected through dropdowns
- Joins are used to display business labels:
  - names, product titles, storage names, role-specific context

### 6.3 CRUD implementation style
- Multi-entity modules use action-driven POST switches
- Entity edit state is controlled through query parameters
- Redirect-after-write pattern prevents duplicate submits

### 6.4 Integrity controls
- SQL FK constraints enforce relational correctness
- SQL CHECK constraints exist for location integrity
- App-level checks reinforce constraints before write operations

## 7. End-to-End Workflow Summary

1. Field Officer logs farmers/fields/harvest and creates input requests.
2. Inventory Manager ensures product master data is correct.
3. Manager creates purchase orders to suppliers with line items.
4. Supplier updates delivery status.
5. Inventory team receives delivered items into storage stock.
6. IoT telemetry is captured from field/storage sensors.
7. System derives environment condition signals from telemetry.
8. QC officer inspects stock and records spoilage/quality condition.
9. Dashboard consolidates all key indicators for management decisions.

---

## 8. Important Assumptions and Implementation Notes

1. Header search input is visual only (disabled), not wired to global search logic.
2. Table pagination is not implemented; lists are fully rendered.
3. Supplier role is scoped for own order updates, while supplier list remains visible in read-focused form.
4. Environment recomputation runs on telemetry create/update events only (no background scheduler).
5. Static mockup HTML files are retained in workspace as references; runtime uses PHP module routes.
6. Some functional domains are grouped into combined pages rather than separate controllers:
   - suppliers + orders + line items in one module
   - managers + storage + stock in one module
   - sensors + telemetry + environment in one module
